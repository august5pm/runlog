# 프로젝트 프롬프트 회고 — 기획 · 디자인 · 개발

> **목적:** 저장소 작업과 대화 이력을 바탕으로, 요청(프롬프트)이 어떤 단계로 이어졌는지 정리하고 프롬프트 작성에 대한 메타 회고를 남긴다.  
> **참고:** 초기 착수 시점의 원문 프롬프트 전체 로그는 이 파일에 전부 보존되어 있지 않으며, **`docs/기획`·`docs/디자인` 문서**, **README**, **현재 코드**, **후속 대화 요약**을 종합해 재구성했다.

---

## 1. 단계별로 본 요청 흐름 (요약)

### 1.1 기획에 해당하는 요청·결과

| 방향 | 내용 |
|------|------|
| 제품 정의 | 러닝 기록 CRUD, 주·월 차트, Google 로그인, Vercel 배포 가능한 토이 범위 |
| 비목표 | 소셜, 웨어러블 자동 연동, 네이티브 앱, 결제 등 명시적 배제 |
| 데이터·API | `User`/`Run` 모델, REST Route Handlers, 본인 데이터만 접근 |
| 문서화 | `기획-러닝기록앱.md`에 MVP(F-01~), 확장(F-06, F-07 날씨 등), 성공 기준·리스크 정리 |

**프롬프트 특징(추정·종합):** 목표와 비목표를 나누고, 1주 스프린트·토이라는 스코프가 문서에 반영되어 있어 이후 요청과 충돌이 적다.

### 1.2 디자인에 해당하는 요청·결과

| 방향 | 내용 |
|------|------|
| IA·내비 | `/dashboard`, `/runs`, `/settings`, 모바일 하단 탭 vs 데스크톱 상단 헤더 |
| 토큰·타이포 | `design-tokens.css` ↔ `globals.css`, Tailwind 매핑 |

**프롬프트 특징:** “어디에 두일지”가 구체화되면서(우측 헤더, 영역 축소) 디자인 문서(`디자인-러닝기록앱.md` v4)와 코드가 같이 갱신됨.

## 2. 프롬프트를 잘 작성했던 점

1. **증상 + 맥락**  
   - 예: “401”, “파싱 오류”처럼 화면/로그에 나온 문구를 주면, 원인 후보(인증 vs 파싱)를 좁히기 쉽다.

2. **레이아웃 의도를 공간 기준으로 설명**  
   - “대시보드 타이틀·인사와 같은 줄 우측”, “영역이 너무 큼”처럼 **위치·밀도**를 말하면 UI 수정 방향이 분명하다.

3. **문서와 코드 동기화 요청**  
   - “현재 코드와 기획·디자인 비교해서 업데이트”는 산출물(`docs/`)을 명확히 해서 이후 온보딩에 도움이 된다.

4. **배포 환경 질문을 분리**  
   - “Vercel에 추가할 게 있나?”처럼 **플랫폼 단위**로 물으면 체크리스트형으로 답하기 좋다.

5. **반복 시 재현 단계**  
   - 재시작 요청 등 짧은 명령은 범위가 작아 실행 오해가 적다.

---

## 3. 프롬프트가 아쉬웠던 점 (개선 여지)

1. **초기 오류 시 스냅샷 부족**  
   - HTTP 상태, 응답 본문 일부(JSON `resultCode`), 사용 중인 키 종류(Encoding/Decoding)가 없으면 추측 수정이 늘어난다.  
   - **개선:** “브라우저/터미널에 보이는 한 줄” 또는 “Network 탭 status”를 같이 적기.

2. **비기능 요구가 나중에 등장**  
   - 날씨는 외부 API 한도·지연·실패 시 UX가 중요한데, 처음부터 “실패 시 어떻게 보일지”가 없으면 기본값으로만 간다.  
   - **개선:** “키 없을 때는 카드 숨김 vs 안내 문구”처럼 정책 한 줄.

3. **문서 업데이트 범위**  
   - “비교해서 업데이트”만 있으면 코드 수정까지 포함인지 문서만인지 모호할 수 있다.  
   - **개선:** “문서만” / “코드까지 기획 반영” 구분.

4. **전체 프롬프트 로그 미보존**  
   - 회고 시 초기 문장을 재구성하게 된다.  
   - **개선:** 중요한 결정은 `docs/` 또는 PR 본문에 남기기.

---

## 4. 앞으로 개선하면 좋은 점 (프롬프트·협업 습관)

| 영역 | 제안 |
|------|------|
| **버그** | 재현 순서, 기대 vs 실제, 로그/스크린 한 줄, 환경(로컬/Vercel, 브랜치) |
| **기능** | 사용자 스토리 1줄 + 비목표 1줄 + “완료 정의”(체크 가능한 문장) |
| **UI** | 와이어 수준: 모바일/데스크톱, 우선순위(필수/후순위) |
| **API 연동** | 공식 문서 링크, 포털에서 신청한 API 이름, 키 종류 |
| **배포** | 사용 호스트 목록(Preview 포함 시 OAuth URI 이슈) |
| **보안** | 시크릿은 채팅에 붙이지 않고 “설정됨/미설정”만 |
